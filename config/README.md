# ApxWiki Config System — Agent Reference

This document explains how the ApxWiki static site is managed. It is written for AI agents that will create articles, update configuration, and run builds. Read this file before making any changes to the wiki.

## Directory layout

```
apxwiki/
  index.html                          ← generated by build.py (do not hand-edit)
  *.html                              ← individual article pages
  config/
    README.md                         ← this file
    site.json                         ← global key-value configuration
    pages.json                        ← page registry (published articles + stubs)
    build.py                          ← build script that assembles everything
    components/
      header.html                     ← nav bar template (injected into every page)
      footer.html                     ← footer template (injected into every page)
```

The rule of thumb: **edit config files and templates, then run `build.py`**. Never hand-edit `index.html` — it is fully generated. Article HTML files can be edited directly for content changes, but their nav, alerts, footer, and CSS are overwritten by the build.

---

## site.json — Global configuration

A flat key-value store for site-wide settings. The build script uses these values in two ways: (1) as `{{key}}` template variables in header.html and footer.html, and (2) as structured data read directly by build.py.

### String keys (available as template variables)

Any top-level key whose value is a string becomes a `{{key}}` placeholder that can be used in `header.html` and `footer.html`. The build script also adds `{{build_date}}` automatically (formatted as "March 7, 2026").

| Key | Purpose | Example value |
|-----|---------|---------------|
| `site_name` | Short brand name used in nav, footer, headings | `"ApxWiki"` |
| `site_title` | Full `<title>` tag for index.html | `"ApxWiki — Appomattox County Encyclopedia"` |
| `site_description` | One-sentence description shown on the index intro | `"A collection of Wikipedia-style..."` |
| `site_tagline` | Shorter tagline used in footer | `"A community encyclopedia of..."` |
| `site_sources` | Attribution line in footer | `"Generated from local newspaper archives..."` |
| `author` | Site author name | `"Nathan Simpson"` |
| `base_url` | Base URL prefix if hosting requires one (empty for flat-file) | `""` |
| `max_width` | Content max-width (informational; actual value is in the CSS) | `"960px"` |

### Structured keys (read by build.py logic)

| Key | Type | Purpose |
|-----|------|---------|
| `alerts` | Array of alert objects | Global alerts shown on every page (see Alerts section below) |
| `stub_categories` | Object `{key: label}` | Display names for stub category groups on index.html |
| `type_labels` | Object `{type: label}` | Human-readable labels for article type codes |

### Adding a new global config value

Add the key to `site.json`. If its value is a string, it is immediately available as `{{key}}` in header.html and footer.html without any code changes. If you need it used in build logic (index generation, article processing), you will need to update `build.py` to read it.

## pages.json — Page registry

This is the central index of every page in the wiki. It has two top-level arrays: `pages` (published articles) and `stubs` (proposed articles that don't exist yet).

### Published page entry

```json
{
  "filename": "Appomattox_Volunteer_Fire_Department.html",
  "title": "Appomattox Volunteer Fire Department",
  "description": "Short description shown on index.html.",
  "type": "organization",
  "status": "published",
  "date_created": "2026-03-04",
  "content_length": 29096,
  "alerts": [
    {
      "type": "info",
      "message": "This article is a first draft and has not yet been reviewed."
    }
  ]
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `filename` | Yes | HTML filename in the apxwiki/ root directory |
| `title` | Yes | Display title used on index.html |
| `description` | Yes | One or two sentence summary for the index listing |
| `type` | Yes | One of: `biography`, `government-topic-history`, `organization`, `notable-event` |
| `status` | Yes | Must be `"published"` to appear on the site |
| `date_created` | Yes | ISO date string (YYYY-MM-DD) |
| `content_length` | Yes | Total file size in bytes; updated by `--update-lengths` |
| `alerts` | No | Optional array of per-page alert objects (see Alerts section) |

### Stub entry

```json
{
  "filename": "Gary_Tanner.html",
  "title": "Gary Tanner",
  "description": "Board of Supervisors member and longtime local government figure.",
  "type": "biography",
  "category": "biographies"
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `filename` | Yes | Intended filename when the article is eventually created |
| `title` | Yes | Display title |
| `description` | Yes | Brief description for the stub listing |
| `type` | Yes | Same type codes as published pages |
| `category` | Yes | Grouping key for the index stub sections; must match a key in `site.json` → `stub_categories` |

Category keys and their display labels (from `site.json`):

- `biographies` → "Biographies — People"
- `events` → "Notable Events"
- `government` → "Government Topic Histories"
- `organizations` → "Organizations and Government Entities"

### Promoting a stub to a published article

When you create a new article HTML file:

1. Remove (or leave) the stub entry from the `stubs` array — build.py will show a checkmark next to any stub whose filename matches a published page.
2. Add a full entry to the `pages` array with `"status": "published"` and all required fields.
3. Run `build.py --update-lengths` to set the correct `content_length`.

## Alerts system

Alerts are config-driven banners rendered inside the `#content` container, immediately after the `<h1>` title on every page. They sit within the page padding and respect the content column width.

### Alert object schema

```json
{
  "id": "ai-disclaimer",
  "type": "warning",
  "message": "Plain text or HTML content for the alert."
}
```

| Field | Required | Description |
|-------|----------|-------------|
| `id` | No | Optional identifier; becomes `id="alert-{id}"` on the HTML element |
| `type` | Yes | Visual style: `warning` (amber), `info` (blue), `success` (green), `danger` (red) |
| `message` | Yes | Alert text content; rendered as inner HTML so basic tags are supported |

### Global alerts vs. per-page alerts

**Global alerts** are defined in `site.json` → `alerts`. They appear on every page including index.html.

**Per-page alerts** are defined on individual page entries in `pages.json` → `pages[n].alerts`. They appear only on that specific article, after the global alerts.

The build script combines them: global alerts first, then per-page alerts, rendered in order as stacked `<div>` elements.

### Common alert use cases

- **AI accuracy disclaimer** (global, type `warning`): The current default. Appears on every page.
- **First draft notice** (per-page, type `info`): Flag an article that hasn't been reviewed yet.
- **Major update notice** (per-page, type `success`): Note that an article was substantially revised.
- **Factual dispute** (per-page, type `danger`): Flag an article with known significant errors pending correction.
- **Temporary site-wide notice** (global, type `info`): e.g., "Several articles are being updated this week."

### Adding or removing alerts

To add a **global alert**, append to the `alerts` array in `site.json` and rebuild.

To add a **per-page alert**, add an `alerts` array to the page entry in `pages.json` and rebuild.

To remove an alert, delete it from the relevant JSON file and rebuild. The build script uses HTML comment markers (`<!-- APXWIKI-ALERTS-START -->` / `<!-- APXWIKI-ALERTS-END -->`) to cleanly replace the alerts block on each build without touching article content.

## Component templates

### header.html

The navigation bar. Currently contains only the ApxWiki brand link back to index.html. Injected above `<div id="content">` on every page.

Supports `{{key}}` placeholders from site.json string values.

```html
<div id="apxwiki-nav">
  <div class="nav-inner">
    <a class="nav-brand" href="index.html">ApxWiki</a>
  </div>
</div>
```

### footer.html

Injected inside `<div id="content">` at the bottom of every page.

Supports `{{key}}` placeholders. Currently uses: `{{site_name}}`, `{{site_tagline}}`, `{{site_sources}}`, `{{build_date}}`.

```html
<div id="apxwiki-footer">
  <b>{{site_name}}</b> — {{site_tagline}}
  {{site_sources}}
  Last updated {{build_date}}.
</div>
```

## build.py — Build script

The build script reads all config files and templates, then generates/updates HTML files. It must be run from the `apxwiki/` root directory.

### Commands

| Command | What it does |
|---------|-------------|
| `python3 config/build.py` | Full rebuild: regenerate index.html and update nav/alerts/footer/CSS on all published articles |
| `python3 config/build.py --index-only` | Rebuild only index.html (skip article updates) |
| `python3 config/build.py --update-lengths` | Recalculate `content_length` for each published page and save to pages.json |
| `python3 config/build.py --dry-run` | Preview what would change without writing any files |

Flags can be combined: `python3 config/build.py --update-lengths --dry-run`

### What the build does to each article

1. **Replaces the `<style>` block** with the canonical shared CSS from `get_shared_css()` in build.py.
2. **Replaces the nav bar** (everything from `<!-- ApxWiki Header` to `<div id="content">`) with the rendered header.html template.
3. **Inserts or replaces the alerts block** inside `#content`, between `<!-- APXWIKI-ALERTS-START -->` and `<!-- APXWIKI-ALERTS-END -->` markers, right after the `<h1>` tag.
4. **Replaces the footer** with the rendered footer.html template.

Article body content (everything between the alerts block and the footer) is preserved untouched.

### What the build does to index.html

Generates it entirely from scratch using:

- Published pages from `pages.json` → sorted alphabetically, with title, description, type label, and character count
- Stubs from `pages.json` → grouped by category, with red links for unwritten articles and checkmarks for stubs that have been published
- Global alerts from `site.json`
- Header and footer from templates
- Stats counts (published articles, proposed stubs)

### CSS management

All CSS is defined in the `get_shared_css()` function inside build.py. This is the single source of truth for styling. Every build overwrites the `<style>` block in every page with this CSS. To change styling site-wide, edit `get_shared_css()` and rebuild.

The CSS includes styles for: layout, typography, nav bar, alerts (all four types), infobox, wikitable, timeline-table, table of contents, references, notes, hatnotes, blockquotes, article-list, stub-list, intro box, stats, footer, and print media rules.

## Workflows

### Creating a new article

1. Write the article HTML file following the apxwiki SKILL.md conventions (article types, infobox, TOC, citations, etc.).
2. Place the file in the `apxwiki/` root directory.
3. Add a published entry to `pages.json` → `pages` array with all required fields.
4. Optionally add per-page alerts if the article needs any.
5. Run: `cd apxwiki && python3 config/build.py --update-lengths`
6. The build will inject the nav, alerts, footer, and CSS into the new article and regenerate index.html with the new listing.

### Updating site-wide configuration

1. Edit `site.json` (change a value, add an alert, etc.).
2. Run: `cd apxwiki && python3 config/build.py`
3. All pages will be updated with the new values on next build.

### Changing the nav bar or footer

1. Edit the template in `config/components/header.html` or `config/components/footer.html`.
2. Run: `cd apxwiki && python3 config/build.py`
3. Every page gets the updated component.

### Adding a page-specific alert

1. Open `pages.json` and find the page entry.
2. Add an `"alerts"` array with one or more alert objects.
3. Run: `cd apxwiki && python3 config/build.py`

### Removing a page-specific alert

1. Delete the alert from the page's `alerts` array in `pages.json` (or remove the entire `alerts` key).
2. Run: `cd apxwiki && python3 config/build.py`
3. The alerts block will be regenerated with only the remaining alerts (global ones will still appear).

### Adding a new stub

1. Add an entry to `pages.json` → `stubs` array with filename, title, description, type, and category.
2. Run: `cd apxwiki && python3 config/build.py --index-only`
3. The stub will appear as a red link on index.html.

### Updating CSS site-wide

1. Edit the `get_shared_css()` function in `config/build.py`.
2. Run: `cd apxwiki && python3 config/build.py`
3. Every page gets the updated stylesheet.

## Important notes

- **Always run build.py from the `apxwiki/` directory.** The script resolves paths relative to itself.
- **Do not edit index.html by hand.** It is fully generated and will be overwritten.
- **Article content is safe.** The build only touches the `<style>` block, nav, alerts block, and footer. Everything else in an article file is preserved.
- **The `{{key}}` template system is intentionally simple.** It only supports flat string replacements from site.json. Nested objects and arrays are not expanded as template variables — they are read by build.py Python logic instead.
- **content_length tracks raw file size in bytes**, not just the article text. It includes HTML markup, CSS, and all injected components. Use `--update-lengths` after any build to keep it accurate.
- **Alert markers are HTML comments.** If an article has never been built with the current system, the first build will insert the alert markers after `</h1>`. Subsequent builds replace between the markers. Do not delete the `<!-- APXWIKI-ALERTS-START -->` / `<!-- APXWIKI-ALERTS-END -->` comments from article files.
